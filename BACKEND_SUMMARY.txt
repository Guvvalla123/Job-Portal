1. WHAT THE BACKEND DOES
The API is an Express 5 job-portal backend for candidates, recruiters, and admins:
MongoDB/Mongoose persistence, JWT sessions with httpOnly refresh cookies, CSRF on
mutations, job listings and applications, companies, notifications, job alerts (with
optional BullMQ email workers and digest scheduling when Redis is configured).

4. DATABASE MODELS

User
  Purpose: Accounts for candidates/recruiters/admins; profile, resume refs, saved jobs, MFA, sessions.
  Key fields: fullName, email (unique, lowercase), password (bcrypt hashed on save, cost 12), role enum,
    profile fields, experience/projects/education subdocs, profileImageUrl/profileImagePublicId,
    resumeUrl/resumePublicId/resumeFileName/resumeSize/resumeUploadedAt, refreshToken (SHA-256 of refresh JWT stored),
    isActive, passwordResetToken/passwordResetExpires, savedJobs [Job], mfaTotpSecretEnc (select:false),
    mfaEnabled, timestamps.
  Indexes: unique on email (schema).
  Relationships: savedJobs → Job; referenced by Job.postedBy, Application.candidate, Company.createdBy, etc.
  Special: toJSON strips password, refreshToken, mfaTotpSecretEnc; comparePassword method.

Job
  Purpose: Job postings linked to company and poster.
  Key fields: title, description, location, employmentType, experienceLevel, minSalary, maxSalary, skills,
    company (Company ref), postedBy (User ref), isActive, isDraft, expiresAt, timestamps.
  Indexes: title, location, employmentType, experienceLevel, skills (indexed), isActive, isDraft, expiresAt;
    text index on title, description, skills, location.
  Relationships: company → Company; postedBy → User; Application.job → Job.
  Special: public listings exclude drafts/expired/inactive per jobRepository filters.

Application
  Purpose: Candidate application to a job with pipeline and interview subdoc.
  Key fields: job, candidate, coverLetter, status (APPLICATION_STATUSES), recruiterNotes, interview
    (scheduledAt, timezone, durationMinutes, notes, status INTERVIEW_STATUSES), timestamps.
  Indexes: unique { job, candidate }; { job, status }; sparse { interview.scheduledAt }.
  Relationships: job → Job; candidate → User.
  Special: one application per (job, candidate) pair.

Company
  Purpose: Employer entity for jobs.
  Key fields: name, website, location, description, logoUrl, createdBy (User ref, optional), timestamps.
  Indexes: { createdBy, createdAt }; text index on name.
  Relationships: createdBy → User; Job.company → Company.

Notification
  Purpose: In-app notifications per user.
  Key fields: user, type enum (APPLICATION_RECEIVED, APPLICATION_STATUS_CHANGED, INTERVIEW_SCHEDULED,
    JOB_ALERT_MATCH, GENERAL), title, message, link, read, readAt, meta (Mixed), timestamps.
  Indexes: { user, createdAt }; { user, read, createdAt }.
  Relationships: user → User.

JobAlert
  Purpose: Candidate job-alert subscriptions with email frequency.
  Key fields: user, keywords[], location, employmentType, salaryMin, frequency (IMMEDIATE|DAILY|WEEKLY),
    isActive, lastSentAt, timestamps.
  Indexes: { user, isActive }; { isActive, frequency }.
  Relationships: user → User.

AuditLog
  Purpose: Immutable audit trail with TTL expiry.
  Key fields: userId, action enum (create|update|delete|login|logout|register|resume_viewed),
    resourceType, resourceId, changes (Mixed), ip, userAgent, timestamps.
  Indexes: userId, action, resourceType, resourceId; TTL on createdAt (expireAfterSeconds from AUDIT_LOG_TTL_DAYS).
  Relationships: userId → User.

indexes.js (startup)
  Purpose: ensureIndexes() — Job.ensureIndexes(), Application.ensureIndexes().

7. AUTH SYSTEM
Access token: JWT HS256, signed with JWT_ACCESS_SECRET; payload includes userId, role, amfa (admin MFA step-up);
  jwtid from crypto.randomUUID() per issue; verified via verifyAccessToken with algorithms pinned to HS256.
  Logout blacklists access tokens by jti in cache (Redis or memory) for remaining TTL.

Refresh token: JWT HS256 with JWT_REFRESH_SECRET; new jti each issue; httpOnly cookie jid_refresh path /api/v1/auth;
  SameSite from COOKIE_SAME_SITE or strict in production / lax in dev; Secure in production.
  Only SHA-256 hash of full refresh JWT string stored on User.refreshToken (authService.hashToken).

CSRF: Double-submit cookie jid_csrf (readable, path /) must match header X-CSRF-Token on unsafe methods;
  exempt paths listed in middlewares/csrf.js (health, ready, metrics, auth bootstrap, reset-password).

RBAC: roles candidate, recruiter, admin (constants/roles.js). requireRole enforces route roles.
  requireAdminPortalAccess: after admin role, loads user mfaEnabled; if mfaEnabled requires JWT amfa === true;
  if env.adminMfaRequired and user not mfaEnabled — 403 MFA_ENROLLMENT_REQUIRED.

config/permissions.js defines PERMISSIONS for job/application/company ownership used by authorize middleware.

8. EMAIL SYSTEM
Primary queue: BullMQ Queue name "email" (queues/emailQueue.js), Redis via ioredis when REDIS_URL set.
  defaultJobOptions: attempts 3, exponential backoff delay 5000 ms, removeOnComplete count 100, removeOnFail count 500.

Job names handled by emailWorker: raw send (to, subject, html|text), JOB_ALERT, APPLICATION_RECEIVED,
  APPLICATION_STATUS_CHANGED or STATUS_CHANGED, INTERVIEW_SCHEDULED.

Fallback when no Redis: addEmailJob in-process direct sendMail for JOB_ALERT, APPLICATION_RECEIVED,
  APPLICATION_STATUS_CHANGED, INTERVIEW_SCHEDULED; queueEmail for legacy "send" jobs; other types log
  "No Redis; typed email job skipped (not implemented for direct fallback)".

emailWorker: DLQ queue "email:dlq" on final failure after attempts; concurrency 5.

Digest: separate queue "job-alert-digest" (workers/jobAlertDigestWorker.js); repeat cron daily 0 8 * * * UTC,
  weekly 0 9 * * 1 UTC; calls jobAlertDigestService.runDigest(DAILY|WEEKLY).

9. SECURITY
Rate limits (config/security.js RATE_LIMITS when rateLimitingEnabled = NODE_ENV production OR RATE_LIMIT_ENABLED true):
  auth: 15 min window, max 100 dev / 10 prod
  strict: 15 min, max 20 dev / 5 prod (forgot-password)
  api: 15 min, max 1000 dev / 200 prod (skips health, ready, metrics paths)
  write/public defined in RATE_LIMITS but only auth, strict, api used in app.js

Validation: Zod middleware on routes; body size 1mb JSON; multer file type/size limits (images 2MB, PDF 5MB).

Sanitization: sanitizeInput strips Mongo operator keys and prototype pollution patterns.

Encryption: passwords bcrypt 12; refresh token hash SHA-256; MFA TOTP secret encrypted (mfaSecretCrypto);
  password reset token stored as SHA-256 of raw token.

Helmet CSP, HSTS, CORS allowlist from CLIENT_URL, compression, HPP whitelist.
