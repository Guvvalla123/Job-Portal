This file is the high-level picture of what exists in the repo today: product,

documents under DOCUMENTATION INDEX at the end.


1. WHAT THIS APPLICATION IS
───────────────────────────
A full-stack job board (Naukri-style): candidates discover jobs and companies,
apply once per job, maintain profiles and PDF resumes, save jobs, get in-app
notifications, and configure job alerts (immediate / daily / weekly). Recruiters
own companies and jobs, manage application pipelines (status, notes, interviews),
view analytics, and stream applicant resumes they are allowed to see. Admins
moderate users, jobs, and companies; see all applications; and read paginated
audit logs.

Data handled includes PII (names, emails, phones, profile text, resumes). There
are no payments in-repo; treat compliance as general PII (see docs/PRIVACY_AND_DATA.md).


2. TECH STACK
─────────────
Backend (backend/):
  Node.js ≥18, Express 5, Mongoose 9, MongoDB
  Auth: JWT access + httpOnly refresh cookie, CSRF double-submit for mutating API calls
  Validation: Zod
  Security: Helmet, CORS (CLIENT_URL allowlist), express-rate-limit, HPP, sanitizeInput
  Files: Cloudinary; uploads: multer (memory), size limits on resume/logo
  Email: Nodemailer; optional Redis + BullMQ queue + dedicated workers
  Observability: Winston (JSON in prod), request IDs, optional Sentry (@sentry/node),
                Prometheus metrics at GET /api/metrics (X-Metrics-Token when set)
  Other: compression, cookie-parser, bcryptjs, opossum (mail circuit breaker)

Frontend (frontend/):
  React 19, Vite 7, Tailwind CSS 4 (@tailwindcss/vite)
  TanStack Query, React Router 7, React Hook Form + Zod, axios (withCredentials)
  UX: Framer Motion, Recharts, sonner, react-helmet-async, react-error-boundary
  Optional: @sentry/react, web-vitals (VITE_ENABLE_WEB_VITALS=true)

Tooling:
  Root: concurrently + wait-for-api for local dev
  CI: GitHub Actions — backend lint/test, frontend lint/test/build, OpenAPI validate,
      E2E (Mongo seed, API, Vite preview, Playwright)
  Deploy blueprints: render.yaml (API + workers), vercel.json (SPA)

4. USER ROLES & MAIN CAPABILITIES
───────────────────────────────
Candidate:  Public job/company browsing, apply (idempotent with optional
            Idempotency-Key header), dashboard, profile CRUD, resume PDF, saved jobs,
            notifications, job alerts.

Recruiter:  Companies CRUD + logo, jobs CRUD (drafts, expiry), analytics/trends,
            applications for own jobs, interview scheduling fields, resume stream
            for own applicants, notifications.

Admin:      Platform stats/trends, user list/toggle/delete/role, job moderation,
            company delete, all applications, audit logs (paginated, TTL on collection).


5. API SURFACE (SUMMARY)
────────────────────────
Base: Versioned JSON REST under /api/v1/...
System (no v1 prefix on router mount): 
  GET  /api/health   — liveness
  GET  /api/ready    — readiness (Mongo connected vs 503)
  GET  /api/metrics  — Prometheus scrape (token gate optional)

Auth: register, login, logout, refresh, me, forgot/reset password, csrf-token.
Jobs, companies, applications (including apply, pipeline, resume stream),
notifications, job-alerts, user profile/resume/saved-jobs/account delete.
Admin: stats, users, jobs, companies, applications, audit-logs.

Full path list: keep using this file’s historical appendix in Postman or
docs/openapi.yaml + docs/API_CONTRACT.md for pagination caps.

Apply flow highlights:
  • POST /api/v1/applications — candidate only; Zod body; audit; idempotency
    middleware when Idempotency-Key is sent; SPA sends key per attempt
    (frontend/src/api/applicationsApi.js).
  • Duplicate apply: unique index (job + candidate); 409 CONFLICT from service
    or 409 DUPLICATE_KEY from Mongo duplicate key handler.


6. SECURITY & PRODUCTION BEHAVIOR (IMPLEMENTED)
──────────────────────────────────────────────
• Trust proxy: Express trust proxy from env (TRUST_PROXY_HOPS; production default
  1 hop when unset) — before rate limit and req.ip-dependent logic (app.js, env.js).
• CORS: allowedHeaders include Idempotency-Key for cross-origin SPA (security.js);
  verified by cors.integration.test.js.
• CSRF: Most mutating /api routes require matching X-CSRF-Token and cookie;
  auth bootstrap routes exempt (middlewares/csrf.js). In tests, CSRF can be enforced
  with CSRF_ENFORCE_IN_TEST=true (CI).
• Rate limiting: Auth routes stricter; blanket /api limiter skips health/ready/metrics
  when enabled (production or RATE_LIMIT_ENABLED).
• Errors: Consistent JSON shape (success, message, code, errors, requestId);
  production hides generic 500 detail; Mongo duplicate key 11000 → 409 DUPLICATE_KEY.
• Sentry: Optional DSN; 5xx capture with request correlation; beforeSend reduces
  request payload/cookie leakage; sendDefaultPii false.
• Idempotency cache: Same Redis/memory store as utils/cache.js — shared across
  instances only when REDIS_URL is set and initRedis() runs on API startup; otherwise
  per-process (documented in docs/DEPLOYMENT.md).
• Pagination caps (Zod): backend/src/validations/paginationLimits.js —
  public jobs list max 50; application lists max 100 (see docs/API_CONTRACT.md).


7. EMAIL, REDIS, WORKERS
────────────────────────
When REDIS_URL is set: BullMQ queue "email"; workers:
  npm run worker:email
  npm run worker:job-alert-digest
Production expectation for reliable mail/alerts: run web + both workers with Redis
(render.yaml lists three services). Without Redis: in-process direct send with
logging and metrics counters (best-effort; not for strict delivery SLAs).
Metrics: email_direct_fallback_* counters; email_queue_jobs gauge by state on scrape
when queue exists on the API process.


8. FRONTEND ROUTING (HIGH LEVEL)
────────────────────────────────
Public: /, /jobs, /jobs/:id, /companies, /companies/:id, legal (/privacy, /terms,
/cookies), /about, /contact. Routes like /careers, /press, /resources/* redirect
to /about so nav does not promise empty placeholders.

Auth: /login, /register, /forgot-password, /reset-password/:token

Protected: /dashboard (role redirect), candidate/recruiter dashboards, admin/*.

Errors: /unauthorized, /500, NotFound for *.

SEO: Client-rendered app; stance documented in docs/SEO_CRAWLING.md.


9. TESTING & QUALITY GATES
──────────────────────────
Backend: Jest — unit/integration (auth, health, ready, CSRF/RBAC flow, CORS preflight,
apply + idempotency, etc.).
Frontend: Vitest (component tests); production build in CI.
E2E: Playwright (smoke + full flow when E2E_FULL=1); CI seeds Mongo via e2e-seed.js.
Root: npm run openapi:validate (OpenAPI file in docs/openapi.yaml).
Dependabot: .github/dependabot.yml for npm and GitHub Actions.

───────────────────────
